# Gogogo

## Overview

把 `gogogo` 视为一种执行模式，而不是某个项目的实现细节。它的职责只有一个：把用户后续描述解释为“按完整闭环连续执行”，并把具体工作委托给最匹配的项目或领域 skill。

不要在这里写仓库特定命令、环境路径、部署主机或业务规则。那些内容应由目标项目自己的 skill 负责。

默认用户画像是单人开发者，不是多人协作团队。可以为了隔离风险在本地使用 worktree、临时分支或实验分支，但它们默认只是本地开发载体，不是线上发布分支；默认也不需要经过 PR 才能合回主线，而是先在真正开始改代码前把本地 `main` 同步到远端最新，再基于这个最新 `main` 开发，最后把要上线的结果集成回 `main`，确认当前分支就是 `main`，然后只从完整 `main` 直接 push / deploy。

发布语义比本地开发语义更严格。完整 `main` 指：本地 `main` 已同步 `origin/main`；所有本次要发布的 worktree、临时分支或实验分支结果已经合入 `main`；`main` 工作区和暂存区干净；最终验证在 `main` 上通过；进入 push / deploy 前再次确认当前分支是 `main`。如果这些条件不成立，不要用 `stash` 或从非 `main` cherry-pick 来抢发版本。

## Workflow

1. 识别 `gogogo` 后面的真实任务，并提炼交付目标、目标仓库、是否涉及 issue、是否涉及部署。
2. 判断描述中是否已经包含 issue 编号、`#N`、issue URL 或其他明确可解析的 issue 引用。
3. 如果已经有 issue 引用，跳过建 issue，直接把该 issue 作为后续实现、review、集成和收尾的追踪主线。
4. 如果没有 issue 引用，先根据后续描述创建一个追踪 issue，再继续后续闭环。
5. 在任何代码修改前，先把本地 `main` 同步到远端最新；默认做法是切到 `main` 后执行 `git pull --ff-only`，或执行项目 skill 规定的更严格等价流程。
6. 若 `main` 工作区存在无关改动、并发修改、分叉状态或其他无法安全直接开发的情况，先从最新 `origin/main` 建立干净 worktree，在该 worktree 中开发、验证和 review；不要为了同步或部署而 stash 这些改动。
7. 若后续要在本地临时分支或 worktree 上开发，确保它是从已同步的 `main` 切出，或先把该分支 rebase/merge 到最新 `main` 后再开始改动。
8. 识别当前任务对应的最具体 skill，并优先调用它来完成项目相关动作。
9. 先复现问题或确认事实，再定位根因和系统性修复点；不要跳到实现。
10. 在不触发阻断条件的前提下，持续推进直到闭环完成，而不是每完成一步就停下来问用户要不要继续。

默认闭环顺序：

1. 建立或确认 issue 追踪主线
2. 同步本地 `main` 到远端最新
3. 复现或确认事实
4. 根因分析与方案约束：说明问题发生在哪一层、为什么现有机制失效、修复应落在哪个通用机制
5. 写失败测试或可复现验证，证明当前行为确实坏了
6. 实现修复或功能
7. 运行相关测试与构建验证
8. 提交前代码审查，并修复阻断问题
9. 再次运行受影响验证，确认 review 修复没有引入回归
10. 提交代码
11. 若当前在本地临时分支或 worktree 分支，先把结果集成回本地 `main`
12. 默认不走 PR 合并主分支；只有用户明确要求时才保留 PR 流程
13. 在 `main` 上重新检查工作区、暂存区、分支名、同步状态和必要验证，确认它是完整 `main`
14. 只从完整 `main` 执行 push，然后部署或发布
15. 若合入 `main` 后发现还有其他主体的并发修改未完成、未验证或待一起发布，停止本次部署并等待该主体统一发布
16. 做上线后验证
17. 若本次通过本地临时分支或 worktree 分支完成，合并完成后删除对应本地开发分支
18. 关闭 issue 或完成收尾动作

## Root Cause Rules

- 修 bug 或质量问题时，必须先完成根因分析；没有根因，不准写生产代码。
- 根因分析至少回答：
  - 失败现象是什么，如何稳定复现或用现有数据证明
  - 坏值、坏排序、坏状态或坏输出从哪里进入系统
  - 现有抽象、算法、配置或数据流哪一层没兜住
  - 为什么修在这里是通用机制，不是只对当前样本贴胶布
- 如果有上游参考实现、旧实现、竞品实现或用户指定参考仓库，先对照其真实机制，再决定修法。
- 如果三次修复尝试都被新反例打穿，停止叠补丁，重新审视架构或抽象边界。

## Anti-Hardcoding Rules

- 默认禁止按具体题面、具体实体名、具体 query 文案、具体客户名、具体测试文件名写分支逻辑。
- 允许写领域通用规则，但必须能描述成可复用模式，例如“参与者列表问题使用 query 锚点附近的枚举证据”，不能是“桃园结义返回刘关张”。
- 任何看起来像硬编码的改动，提交前必须满足三件事：
  1. 有一个失败测试覆盖原始问题
  2. 至少有一个反例或干扰样本证明不是只贴当前 case
  3. 文档或注释说明它对应的通用机制
- 如果只能临时硬编码止血，必须在 issue / 注释 / 文档中标记为临时 workaround，并创建后续清理项；不能把它伪装成架构优化。

## Review Gate

- `git add`、`git commit`、`git push` 之前必须完成提交前代码审查；这是硬门槛，不是建议。
- 优先使用项目或会话中可用的 code review skill / 工具；没有可用工具时，做显式自审并记录结论。
- review 必须至少检查：
  - 是否存在硬编码、样本特化、测试投机或只修症状
  - 根因分析是否能解释改动位置
  - 测试是否先覆盖失败，再覆盖成功和关键反例
  - 文档、配置、CLI/API 行为是否与代码一致
  - 是否误提交本地数据、构建产物、密钥、日志或无关文件
  - 是否有未处理的脏工作区改动会被混进提交
- review 发现阻断问题时，先修复，再重跑受影响验证，然后重新过 review gate。
- 最终回复要说明 review gate 的结论；如果因为缺少工具只能自审，也要明说。

## Delegation Rules

- `gogogo` 只负责定义节奏与阶段顺序，不负责替代项目 skill。
- 发现仓库里已经有更具体的 skill 时，优先委托给它。例如：
  - GitHub issue 闭环优先用 `$github-issue`
  - 某个项目已有专用排障/部署 skill 时，优先用该项目 skill
- 若没有更具体的 skill，再退回通用工程流程自行执行。
- 若用户同时显式点名了某个 skill，优先服从那个 skill 的领域规则，`gogogo` 只补充“不要中途停、按闭环跑完”的执行语义。
- 若项目 skill 没有更严格的代码修改前同步规则，默认强制在首次代码修改前同步本地 `main`；不要在落后于远端 `main` 的基线上开始实现。
- 若项目 skill 没有更强的 Git 约束，默认以 `main` 作为唯一线上推送与发布分支；本地临时分支只用于开发过程，不直接承担线上发布语义。
- 若用户没有明确要求 PR、评审门禁或远端协作分支，默认不要额外创建“先提交再通过 PR 合 main”的流程。
- `gogogo` 自带提交前 review gate；即使不走 PR，也不能跳过本地代码审查。

## Release Rules

- 发版本、部署或任何线上发布动作只能从完整 `main` 发起。
- 完整 `main` 必须同时满足：已同步 `origin/main`；包含本次所有待发布 worktree、临时分支或实验分支结果；工作区和暂存区干净；最终验证在 `main` 上通过；进入 push / deploy 前当前分支确认为 `main`。
- 如果主工作区的 `main` 因无关改动、并发修改、分叉或未完成状态而不满足条件，先建立干净 worktree 开发和验证，不要用 `stash` 把主工作区临时伪装成可发布状态。
- 不要从非 `main` 分支 cherry-pick 一部分提交来组成发布内容。默认通过 merge、rebase 或 squash merge 把完整结果带回 `main`；即使用户明确要求用 cherry-pick 搬运本地补丁，最终仍必须回到完整 `main` 并重新验证后才能发布。
- 如果 worktree 验证通过并合入 `main` 后，发现 `main` 还包含其他主体的并发修改、未完成验证或待一起发布的内容，不抢发版本；停止本次部署，记录本次工作已合入或已验证到哪里，等待该主体统一部署。

## Stop Rules

只有在以下情况才中途停下并向用户确认：

- 缺少必要权限、凭据或账号状态，且无法自行补齐
- 涉及不可逆或高风险动作，例如会删除生产数据、强制回滚、破坏性重置
- 缺少关键事实或样本，且无法通过仓库、日志、远端环境或 issue 自行获取
- 本地存在未提交改动、分支分叉或其他状态，导致无法安全把 `main` 同步到远端最新，且又无法通过独立 worktree、额外提交或其他不污染发布主线的方式自行化解
- 继续部署会明显打断未完成任务、在线流量或恢复语义，而项目 skill 又没有给出安全处理规则
- 用户的真实意图与已有 issue、代码、线上事实明显冲突
- 当前工作仍停留在本地临时分支，但又准备直接 push / deploy，且无法自行安全地集成回 `main`
- 准备发布或部署时，`main` 不满足完整 `main` 条件，且无法在不接管其他主体并发修改的前提下自行修复
- 根因仍不清楚，只能靠硬编码或样本特化继续推进
- 提交前 review gate 发现阻断问题且短时间内无法修复

除这些阻断条件外，不要把 `gogogo` 当成“先做一部分再汇报”的指令。

## Git Rules

- 默认把 `main` 视为唯一线上分支。
- 在任何代码修改之前，必须先同步本地 `main` 到远端最新；默认顺序是先切到 `main`，再执行 `git pull --ff-only`，或执行项目 skill 规定的更严格等价流程。
- 若后续在临时分支或 worktree 上开发，必须基于这个已同步的 `main` 创建，或先把现有分支安全地 rebase/merge 到最新 `main`，再开始编辑代码。
- 允许在本地使用 worktree、临时分支、实验分支或 issue 分支完成实现、测试和 review。
- 这些本地分支默认不直接推送到线上作为交付结果，除非用户明确要求保留远端分支、PR 流程或团队协作分支。
- 默认不采用“先把开发分支推上远端，再通过 PR 合并到 `main`”的交付方式。
- 默认做法是：本地开发完成后，先把完整结果集成回本地 `main`，在 `main` 上完成最终验证，再从完整 `main` push / deploy。
- 不要在尚未同步的 `main` 或基于陈旧 `main` 切出的分支上直接开始改代码。
- 在任何 `git push` 之前，必须先显式检查当前分支名。
- 在任何 `git commit` 之前，必须先确认 review gate 已通过、验证已通过、暂存区只包含本次交付相关文件。
- 若当前分支不是 `main`，先把改动集成回 `main`，再从 `main` 执行 push。
- 若 `main` 工作区已有无关或并发改动，不要为了部署执行 `git stash`；改用干净 worktree 完成本次任务，或在无法隔离时停止并说明阻断。
- 禁止用从非 `main` cherry-pick 出来的局部内容直接发布；发布产物必须对应完整 `main`。
- 若本次使用了本地临时分支或 worktree 分支，结果合并进 `main` 后必须删除对应本地开发分支；不要长期残留已经完成交付的本地分支。
- 若项目 skill 有更严格的主线要求，服从更严格的规则；否则按这里执行。

## Decision Rules

- 默认用户画像是单人开发，不假设存在必须保留长期 feature branch 的团队流程。
- 若没有明确相反要求，目标主线同时指向本地与远端 `main`。
- 若准备开始改代码，先验证本地 `main` 已经同步到远端最新；若没有，先同步再动手。
- 允许为了隔离变更而创建本地 worktree 或临时分支，但闭环交付的 push / deploy 必须从 `main` 发起。
- 若已经在临时分支完成修复，先通过 merge、rebase 或 squash merge 把完整结果带回 `main`，再继续 push 与部署。
- 若主工作区已有并发修改，优先建立干净 worktree；不要为了发布清空、stash 或接管这些并发修改。
- 若用户没有明确要求 PR 流程，默认不额外创建 PR 来合并 `main`；本地把结果并回 `main` 后直接从 `main` 推进后续动作。
- 若临时分支上的改动已经成功集成到 `main`，默认要删除该本地分支；除非用户明确要求继续保留。
- 进入 push 阶段时，不要凭印象假设自己“应该在 main”；必须显式检查当前分支。
- 进入 deploy 阶段时，如果发现 `main` 包含其他主体尚未完成的并发修改，则等待该主体完成部署，不要单独挑出本次改动发布。

## Examples

- `gogogo 修复 #123`
- `gogogo https://github.com/org/repo/issues/45`
- `gogogo 这个导入任务会把手机号列识别错，帮我从建 issue 到上线全跑完`
- `gogogo 把这个需求直接做完并发版，不要每一步都停下来确认`
