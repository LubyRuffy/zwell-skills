# Linux Hot Restart

把“热重启”理解为“稳定入口不下线，先拉起新实例、验证通过后再切流并下线旧实例”，而不是简单执行 `systemctl restart`。

优先解决单机 Linux 服务的零停机发布问题。默认首选 `systemd + 本机反向代理 + blue/green 双 slot`，如果项目本来就运行在容器或编排环境里，则优先复用现有容器/LB 的滚动或蓝绿能力，而不是重复造一套控制面。

## 何时使用

- 当前部署方式是“停服务 -> 覆盖二进制 -> 启服务”，会打断在线请求。
- 服务是长期驻留进程，需要在 Linux 主机上无感升级。
- 希望发布流程具备健康检查、切流验证、失败回滚和旧实例延后下线能力。
- 需要把历史的单进程直绑端口模式迁移为稳定代理入口模式。

## 方案选择

- 单机、systemd、裸二进制服务：优先用 `nginx 或其他支持 reload 的本机代理 + blue/green 双 slot`。
- 单机、Docker/Compose 服务：优先保留稳定代理入口，应用实例改为双容器或双 compose service，健康后再切流。
- 已有 Kubernetes、Nomad、ECS、Swarm 等编排：优先用编排器原生 rolling update / blue-green，不要再叠一层自定义主机切流。
- 应用本身已经支持 FD 继承或 socket activation：只有在现有实现成熟且团队能长期维护时，才考虑真正的进程内热重启；否则仍优先代理 + 双实例。

## 核心原则

1. 稳定入口必须独立于业务实例，不能让真实应用直接占住对外服务端口。
2. 新实例必须先在空闲 slot 启动，并通过真实 readiness 检查。
3. 切流只允许发生在“稳定入口已经确认可把请求转给新实例”之后。
4. 旧实例只能在稳定入口验证成功后再下线。
5. 任一步失败，都要保留旧流量路径并清理失败候选实例。
6. readiness 不能只看进程是否活着，必须命中真正能代表可服务状态的健康接口。

## 标准流程

1. 识别稳定入口、当前实例、健康检查接口和可用的第二个 slot。
2. 若当前还是“业务进程直绑稳定端口”，先设计一次性迁移：新增代理层，旧模式仅作为首迁兜底。
3. 为双 slot 分配固定本机端口，例如 `blue`、`green` 各占一个 localhost 端口。
4. 发布新版本到版本化目录，不直接覆盖正在运行的旧实例目录。
5. 启动空闲 slot，并循环等待健康检查成功。
6. 生成新的代理 upstream 配置和 active-slot 状态文件。
7. 先做代理配置校验，再执行 `reload` 切流，不做硬重启。
8. 通过稳定入口再次验证，确认请求已经命中新 slot。
9. 验证通过后，停止旧 slot；若失败，恢复旧 upstream/状态并停掉候选 slot。

## 关键实现要求

- 使用版本化 release 目录和稳定软链，避免原地覆盖可执行文件。
- active slot 必须有持久化状态文件，重启后可恢复当前拓扑。
- 代理配置里最好暴露一个可验证的响应头或其他低成本信号，用于确认稳定入口已切到新 slot。
- 健康检查等待窗口要按真实启动耗时设置，通常应为分钟级，而不是 5 到 10 秒。
- 首次从 legacy 单进程迁移时，要基于“当前正在运行的真实进程”判断模式，而不是只看 unit 文件内容。
- 首迁前应备份 legacy unit，迁移失败时能快速恢复。

## 失败处理

- 新 slot 启动失败：保持旧流量不变，停止候选 slot，发布失败。
- 代理配置校验失败：恢复旧 upstream/状态，停止候选 slot。
- 代理 reload 失败：恢复旧 upstream/状态，重新加载旧配置，停止候选 slot。
- 稳定入口验证失败：恢复旧入口指向；如果这是首次迁移，还要恢复 legacy unit 并重新拉起旧服务。

## 交付物

- 稳定入口代理配置
- 双 slot `systemd` 模板或对应容器编排模板
- active-slot / upstream 等运行时状态文件约定
- 发布脚本中的启动、校验、切流、回滚逻辑
- 使用者文档中的部署流程与回滚说明
